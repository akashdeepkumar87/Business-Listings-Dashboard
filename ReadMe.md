## рҹҡҖ Local Setup

### Prerequisites

- Python 3.11+
- Node.js 18+
- A [Railway](https://railway.app) account (free tier)

---

### 1. Clone the repository

```bash
git clone https://github.com/YOUR_USERNAME/business-dashboard.git
cd business-dashboard
```

---

### 2. Database вҖ” Railway MySQL

1. Go to [railway.app](https://railway.app) вҶ’ **New Project вҶ’ Deploy MySQL**
2. Click the MySQL service вҶ’ **Variables** tab
3. Copy the **public proxy** credentials (not `mysql.railway.internal`):
   - `RAILWAY_TCP_PROXY_DOMAIN` вҶ’ your `DB_HOST`
   - `RAILWAY_TCP_PROXY_PORT` вҶ’ your `DB_PORT` (5-digit number)
   - `MYSQLUSER`, `MYSQLPASSWORD`, `MYSQLDATABASE` as-is

---

### 3. Scraper вҖ” collect + load data

```bash
cd scraper
pip install -r requirements.txt

copy .env.example .env   # then fill in Railway credentials

python scraper.py        # fetches OSM data + generates mock fill вҶ’ listings.csv
python load_to_db.py     # inserts all 661 records into Railway MySQL
python db_dump.py        # exports dump/listing_master.sql for submission
```

---

### 4. Backend вҖ” FastAPI

```bash
cd backend
pip install -r requirements.txt

copy .env.example .env   # same Railway credentials

uvicorn main:app --reload --port 8000
```

Interactive API docs: **http://localhost:8000/docs**

#### Endpoints

| Method | Endpoint                   | Description                                |
| ------ | -------------------------- | ------------------------------------------ |
| `GET`  | `/health`                  | DB connectivity check + total count        |
| `POST` | `/listings/bulk-insert`    | Bulk insert via JSON body                  |
| `GET`  | `/listings`                | Paginated listings (city/category filters) |
| `GET`  | `/dashboard/city-wise`     | Business count grouped by city             |
| `GET`  | `/dashboard/category-wise` | Business count grouped by category         |
| `GET`  | `/dashboard/source-wise`   | Business count grouped by source           |

---

### 5. Frontend вҖ” React

```bash
cd frontend
npm install

copy .env.example .env   # set VITE_API_URL=http://localhost:8000

npm run dev
```

Open: **http://localhost:5173**

> The FastAPI backend must be running before starting the frontend.

## вҳҒпёҸ Production Deployment

### Backend вҶ’ Render

1. Go to [render.com](https://render.com) вҶ’ **New вҶ’ Web Service**
2. Connect your GitHub repo вҖ” Render auto-detects `render.yaml`
3. Add environment variables under **Environment**:

```
DB_HOST        = <Railway public proxy domain>
DB_PORT        = <Railway public port>
DB_USER        = root
DB_PASSWORD    = <your password>
DB_NAME        = railway
ALLOWED_ORIGIN = https://your-app.vercel.app   вҶҗ add after Vercel deploy
```

4. Deploy вҶ’ note your URL: `https://your-app.onrender.com`

---

### Frontend вҶ’ Vercel

1. Go to [vercel.com](https://vercel.com) вҶ’ **New Project** вҶ’ import repo
2. Set **Root Directory** to `frontend`
3. Add environment variable:

```
VITE_API_URL = https://your-app.onrender.com
```

4. Deploy вҶ’ note your URL: `https://your-app.vercel.app`

5. Go back to Render вҶ’ add `ALLOWED_ORIGIN = https://your-app.vercel.app` вҶ’ redeploy

## рҹ“Ҡ Data Collection Approach

### Primary вҖ” OpenStreetMap Overpass API

- Queried real business data across **10 Indian cities**: Mumbai, Delhi, Bangalore, Chennai, Hyderabad, Pune, Kolkata, Ahmedabad, Jaipur, Surat
- Covered **10 business categories**: Restaurant, Hospital, Hotel, School, Salon, Gym, Pharmacy, Bank, Supermarket, Clinic
- Each city queried via geographic bounding box вҖ” returns real OSM nodes with name, phone, address tags
- No API key required вҖ” fully public and free
- Polite crawling: 3-second delay between city queries

### Fallback вҖ” Mock Data Generator

- Fills the gap when OSM returns fewer than 600 records
- Realistic Indian business names per category, valid `+91-XXXXXXXXXX` phone format, real area names
- Final dataset: **661 records** across 10 cities, 10 categories

## рҹ—„пёҸ Database Schema

```sql
CREATE TABLE listing_master (
    id            INT AUTO_INCREMENT PRIMARY KEY,
    business_name VARCHAR(255)  NOT NULL,
    category      VARCHAR(100)  NOT NULL,
    city          VARCHAR(100)  NOT NULL,
    address       TEXT,
    phone         VARCHAR(50)   DEFAULT NULL,
    source        VARCHAR(100)  NOT NULL,
    created_at    DATETIME      DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_city     (city),
    INDEX idx_category (category),
    INDEX idx_source   (source)
) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

## вҡ пёҸ Challenges Faced

### 1. Web scraping restrictions (403 blocked)

**Problem:** Justdial, Sulekha, and Google Maps all returned HTTP 403 when accessed via Python `requests`. These platforms actively block bots.

**Solution:** Switched to the OpenStreetMap Overpass API вҖ” free, public, and legally accessible. Supplemented with a mock data generator to exceed the 500-record requirement, as permitted by the assignment brief.

### 2. Railway internal vs public hostname

**Problem:** Railway's internal hostname (`mysql.railway.internal`) only works from within Railway's own network, causing connection failures from local machines and Render.

**Solution:** Used the public TCP proxy credentials (`RAILWAY_TCP_PROXY_DOMAIN` + `RAILWAY_TCP_PROXY_PORT`) for all external connections.

### 3. Pagination without total count

**Problem:** The original `/listings` endpoint returned a plain list, so the frontend couldn't show "Page 2 of 34" or correctly disable Next/Previous buttons.

**Solution:** Added a `COUNT(*)` query alongside the data query, returning a `PaginatedListingsResponse` with `total`, `total_pages`, `has_next`, and `has_prev`.

### 4. Phone number inconsistency

**Problem:** OSM data contained phone numbers in many formats вҖ” some with country codes, some without, with varying separators.

**Solution:** Added a `_normalize_phone()` field validator in Pydantic models to standardise all numbers to `+91-XXXXXXXXXX` format before storing.

### 5. CORS in production

**Problem:** After deploying the backend to Render, the Vercel frontend was blocked by CORS because `localhost` origins were hardcoded.

**Solution:** Added an `ALLOWED_ORIGIN` environment variable on Render. The backend dynamically builds its allowed origins list from this variable, keeping local dev working while allowing the production Vercel URL.
